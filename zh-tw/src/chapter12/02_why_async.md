# 為什麼需要 `async`

## 本集目標

上一集已經跑過一個 async server，這一集回答一個更根本的問題：**為什麼要有 `async`？什麼時候才用得上它？** 並把它和你已經熟悉的 OS thread 做對比。

## 概念說明

### `async` 是為「大量等待」設計的

`async` 最適合**花很多時間在「等」的程式**——而且等的是**外部的東西**，不是 CPU 在算：

- 網路伺服器：等客戶端送請求、等回應送出去
- 爬蟲：等一個又一個網站回應
- 資料庫查詢：送出 query，等資料庫回來
- 聊天室、即時通訊：大部分時間都在等下一則訊息
- 各種背景工作：等檔案、等計時器、等別的服務

這些程式的共通點是：**大多數時間不是在算，而是在等**。一個連線可能 99% 的時間都卡在「等對方回話」。

### 對比 OS thread：thread 很吃記憶體

你可能會說：「等的話，每個連線開一條 thread 去等不就好了？」可以，但**不划算**。每條 OS thread 都要分一塊**不小的 stack**（常見預設是 MB 等級的記憶體），作業系統還要花力氣在這些 thread 之間切換。

一台只服務幾十條連線的伺服器，開幾十條 thread 沒問題；但一台要同時撐**一萬條**連線的伺服器，開一萬條 thread 就是上 GB 的記憶體、外加沉重的排程成本。問題是這一萬條 thread 幾乎都在**睡覺等資料**——花大量記憶體養一堆「在等的 thread」，非常浪費。

`async` 的做法不一樣：它不為每個「等待」開一條 thread，而是讓**少少幾條 thread**輪流去推進那些「現在剛好可以動」的工作；正在等的工作只是一份很小的狀態，不佔一整條 thread。所以同樣的機器，async 能撐的併發連線數遠遠多於 thread-per-connection。

### `async` 不是讓 CPU 算得更快

這是最常見的誤會，要講清楚：**`async` 不會讓你的計算變快。** 如果你的程式是 CPU 密集（一直在算、不太需要等），async 幫不上忙——那種情況該用的是多執行緒平行運算。

`async` 真正做的事是：**在「等待」的那段時間，把執行緒讓出來給別的工作用。** 它優化的是「等待」，不是「計算」。一句話：

> `async` 不是讓 CPU 跑得更快，而是讓**等待的時候不要白白卡著一條執行緒**。

所以判斷準則很簡單：你的程式**瓶頸在等 I/O，還是在算**？等 I/O（而且量很大）→ async 很適合；純計算 → async 沒幫助。

## 重點整理

- `async` 適合**大量等待 I/O** 的程式：網路伺服器、爬蟲、資料庫、聊天室、背景工作
- 對比 OS thread：每條 thread 都要一塊不小的 stack，**很吃記憶體**；一萬條等待中的連線開一萬條 thread 非常浪費
- `async` 用少少幾條 thread 輪流推進「現在能動」的工作，等待中的工作只是一小份狀態，不佔整條 thread
- `async` **不是讓 CPU 算得更快**，而是讓「等待期間」能把執行緒挪去做別的事；瓶頸在等 I/O 才用得上它，純計算用不上
