# 為什麼需要 async

## 本集目標

搞懂 async 到底解決什麼問題，以及它和多執行緒的差別。

## 概念說明

### async 不會讓計算變快

先破除一個最常見的誤會：**async 不會讓你的程式算得比較快**。如果你的程式是在做一件很耗 CPU 的事——例如計算一張圖片每個像素、或是跑一個很大的迴圈做數學運算——那 async 一點忙都幫不上。CPU 就是要花那麼多時間算完，這跟要不要用 async 無關。

async 解決的是另一種完全不同的問題：**等待**。

### 兩種「忙」

程式的「忙」其實分兩種，分清楚這兩種，就懂 async 為什麼存在了。

第一種叫 **CPU-bound（受限於計算）**：程式真的在拚命算，CPU 一直全速運轉。例如影片轉檔、密碼破解、科學模擬。

第二種叫 **I/O-bound（受限於輸入輸出）**：程式大部分時間其實在**等別人**。等網路另一頭把資料傳過來、等硬碟把檔案讀出來、等資料庫回覆查詢、等使用者按鍵盤。這段等待的時間，CPU 其實閒著沒事幹。

async 專門對付的是第二種。網路伺服器、網路爬蟲、聊天室、需要打很多外部 API 的後端服務——這些都是 I/O-bound，它們的瓶頸不是算不夠快，而是「在等」的時間太多。

### 用多執行緒也可以等，但是很貴

你可能會問：第 8 章不是教過多執行緒嗎？要等的時候，多開一條執行緒去等不就好了？

確實可以，而且對「只需要同時等幾件事」的程式來說，這樣完全沒問題。但問題出在**量**。每開一條作業系統執行緒（OS thread），都要付出代價：

- 每條執行緒都要分配一塊不小的記憶體（常見是 MB 等級）當它的 stack。
- 作業系統要在這些執行緒之間切換（context switch），執行緒越多，切換的開銷越大。

如果你要同時服務一萬個連線，開一萬條執行緒，光是記憶體就吃掉好幾 GB，而且大部分執行緒其實都只是卡在那裡等而已——把一整條執行緒拿去「等」，太奢侈了。

### async 的做法：一條執行緒招呼很多工作

async 換了個思路。它不再「一個等待配一條執行緒」，而是讓**少少幾條執行緒**（甚至一條）去輪流招呼**成千上萬個工作**。

關鍵就在上一集那個 `.await`：每當一個工作走到要等待的地方（`.await`），它就主動讓出執行緒，說「我要等一下，這段時間你先去忙別人的」。執行緒於是跳去推進另一個工作，等到原本那個工作等的事情好了，再找機會回來繼續它。

因為「等待」不再佔住執行緒，你就可以用很少的資源同時招呼海量的連線。這就是為什麼網路伺服器特別愛用 async。

### 一個比喻

想像一間餐廳。

多執行緒的做法，是**每桌客人都配一位專屬服務生**。客人在看菜單想很久（等待）的時候，那位服務生就只能站在旁邊等他，什麼別的事都不能做。要服務一百桌，就得請一百個服務生。

async 的做法，是**一位服務生招呼很多桌**。他幫 A 桌點完餐，A 桌開始吃（等待），他不會站著看 A 桌吃，而是立刻轉去招呼 B 桌、C 桌。哪一桌需要服務了，他再過去。一個服務生就能照顧一整區，因為「客人在吃」「客人在想」這些等待的時間，他都拿去服務別桌了。

`.await` 就是那個「我這桌現在要等了，服務生你先去忙別桌」的訊號。

## 重點整理

- async **不會讓計算變快**，它解決的是「等待」的問題
- CPU-bound：程式在拚命算，async 幫不上忙
- I/O-bound：程式大部分時間在等網路、硬碟、資料庫等——這才是 async 的主場
- 多執行緒也能處理等待，但「一個等待配一條 OS 執行緒」在數量大的時候太貴（記憶體、切換開銷）
- async 讓少數幾條執行緒輪流招呼大量工作，等待時就把執行緒讓給別的工作用
- 餐廳比喻：多執行緒是一桌配一個服務生，async 是一個服務生招呼很多桌
