# `async` 是一種 effect

## 本集目標

把 `.await` 和你已經很熟的 `?` 看成**同一種東西**，理解這種「作用」怎麼運作，以及它最後一定要怎麼「落地」回到普通世界。

## 概念說明

### 兩個小尾巴，其實是同一種東西

先把兩段你都看過的程式擺在一起：

```rust,noplayground
let n: i32 = "42".parse()?;       // ? 這個尾巴
let data = fetch_data().await;    // .await 這個尾巴
# fn main() {}
```

`?` 和 `.await` 都是黏在一個運算式**後面的小尾巴**。表面上它們好像只是「把值拿出來」，但每一個其實都偷偷夾帶了一條規矩。要看懂這條規矩，得先看清楚：這兩個尾巴前面的東西，根本不是赤裸的值。

### 先看「被包起來的值」

- `"42".parse::<i32>()` 給你的不是 `i32`，而是 `Result<i32, _>`——「一個 i32，**或者**一個錯誤」。
- `fetch_data()` 給你的不是 `String`，而是一個 future——「一個 String，**只是還沒到**」。

看出共通點了嗎？你手上拿到的都不是**赤裸的值**，而是**被包進一個有特殊規矩的世界裡的值**：

- **Result 世界**的規矩：這個值隨時可能「其實是個錯誤」。
- **Future 世界**的規矩：這個值隨時可能「還要再等一下」。

兩個世界長得不一樣，但形狀一模一樣：都是「一個值，外面裹著一層自己的規矩」。

### 尾巴的工作：把值拉出來，規矩自動幫你處理

如果沒有這兩個尾巴，在這種「包起來的世界」裡做事會很痛苦——每個 `Result` 都要手動 `match`、每個 future 都要手動去 poll，程式很快就變成一團巢狀的縮排地獄。

`?` 和 `.await` 就是來救你的。它們做的是同一件事：

> **把值從世界裡「拉出來」給你直接用，而那個世界的規矩，由尾巴自動幫你串好。**

- `?` 讓你把 `Result<i32, _>` 當成 `i32` 一路用下去；「萬一其實是錯誤」這條規矩，它自動處理（短路、從整個函數提早 return）。
- `.await` 讓你把 future 當成它裡面的值一路用下去；「萬一還沒好」這條規矩，它自動處理（先暫停、把執行緒讓出去，好了再從這裡接著跑）。

最神奇的地方是：你寫出來的程式，長得像一條普通的、一行接一行的直線程式——

```rust,noplayground
# fn parse() -> Result<i32, std::num::ParseIntError> {
let a: i32 = "10".parse()?;
let b: i32 = "20".parse()?;
# Ok(a + b)
# }
```

——但它背後其實在**一步步把這些「包起來的值」串成一條計算**。每一個 `?` / `.await`，就是串接時「套用世界規矩」的那道接縫。你只管照直覺把值接下去寫，規矩在接縫處被悄悄處理掉。`?` 和 `.await` 之所以好用，正是因為它們讓你能在這種包起來的世界裡，寫出和平常一樣自然的程式。

### 串接是有代價的：`async` 會大改函數的長相

`?` 串接的那條規矩很單純（要不要提早跳出去），所以編譯出來的程式和你寫的差不多。`.await` 的規矩複雜多了——「跑到一半暫停、之後再從同一個地方接著跑」——要做到這件事，編譯器會把你的函數體**整個改寫**成一台能暫停與恢復的機器（這台機器叫狀態機，第 15 集會拆解）。你寫的是一條直線程式，背後卻被沿著每個 `.await` 切成好幾段。所以記住：**`async` 不只是加個關鍵字，它會對整個函數體動大手術。**

### 你只能在世界「裡面」把值拉出來：傳染

這裡有個關鍵限制：你要套用某個世界的規矩，你的函數本身就得**待在那個世界裡**。

- 要用 `?` 短路成錯誤，你的函數得回傳 `Result`（你得在 Result 世界裡，才有「錯誤」可短路）。
- 要用 `.await` 暫停，你的函數得是 `async`（你得在 Future 世界裡，才有「暫停」可言）。

所以一個普通（同步）函數，**不能**直接 `.await` 一個 future：

```rust,ignore
async fn fetch_data() -> String {
    String::from("資料")
}

fn normal_function() {
    let data = fetch_data().await; // 編譯錯誤！普通函數不在 Future 世界裡，不能 .await
}
```

你可以「呼叫」`fetch_data()` 拿到一個 future（第 3 集），但要 `.await` 它，你所在的函數自己就得是 `async`。於是這層「色彩」一路往上傳染，把呼叫鏈都染成 async。`?` 也一樣會往上傳染（要求一路上的函數都回傳 `Result`）。

### 最後一定要「落地」：把世界結算掉

傳染不可能無止盡。一個「包起來的計算」不能永遠飄在世界裡，總得在某個地方**被結算回普通世界**——變成一個普通的值、變成一個真的會跑的程式。而結算的方式，剛好有**兩條路**，兩個世界完全平行。

先看你熟悉的 Result 世界，兩條路是：

1. **一路頂到最上面，交給邊界**：寫 `fn main() -> Result<(), E>`。你的 `main` 整個待在 Result 世界裡，`?` 一路往上短路，最後由語言／runtime 在程式的邊界幫你結算（印出錯誤、設定離開碼）。你本人從沒親手攤開那個 `Result`，你把整包交給了邊界。
2. **自己當場結算**：在某個地方寫 `match result { Ok(v) => ..., Err(e) => ... }`，親手把 `Result` 攤開、兩種情況都處理掉。處理完，你就回到了普通世界。

Future 世界，**一模一樣的兩條路**：

1. **一路頂到最上面，交給邊界**：寫 `#[tokio::main] async fn main()`。你的 `main` 整個待在 Future 世界裡，最後由 runtime 在邊界把它推到完成。——這對應到「回傳 `Result` 的 `main`」。
2. **自己當場結算**：在普通的同步程式裡寫 `runtime.block_on(async { ... })`，親手把 future 推到完成、拿回一個普通的值。——這對應到 `match`。

把它排成一張表最清楚：

```text
          在世界裡傳播      落地法一：頂到邊界（交給框架）      落地法二：自己當場結算
Result      ?              fn main() -> Result<...>          match
Future     .await          #[tokio::main]                    block_on
```

深一層看：`#[tokio::main]` 和「回傳 `Result` 的 `main`」是同一種精神——**把整包作用頂到程式的邊界，讓框架去結算**；`block_on` 和 `match` 是另一種精神——**你在普通世界的某一點，親手把作用結算成一個普通的值**。

這也正好回答了「同步程式要怎麼跨進 async」這個問題：`block_on` 就是那座橋——它讓你在普通（同步）程式裡，把一個 future 當場結算成普通的值。而 `#[tokio::main]` 其實只是個語法糖，幫你的 `main` 在邊界自動包了一層 `block_on`（細節第 21 集）。

## 範例程式碼

把兩個世界的「兩條落地路線」並排看，平行關係會很清楚。

Result 世界——同一個 `parse_both`，兩種結算方式：

```rust,ignore
fn parse_both() -> Result<i32, std::num::ParseIntError> {
    let a: i32 = "10".parse()?; // ? 把值拉出來，錯誤自動短路
    let b: i32 = "20".parse()?;
    Ok(a + b)
}

// 落地法一：交給邊界——main 自己待在 Result 世界
fn main() -> Result<(), std::num::ParseIntError> {
    let sum = parse_both()?;
    println!("{}", sum);
    Ok(())
}

// 落地法二（擇一）：自己當場 match 結算
// fn main() {
//     match parse_both() {
//         Ok(sum) => println!("{}", sum),
//         Err(e) => println!("壞了：{}", e),
//     }
// }
```

Future 世界——同一個 `fetch_both`，兩種結算方式，跟上面長得一模一樣：

```rust,ignore
async fn fetch_both() -> i32 {
    let a = fetch(10).await; // .await 把值拉出來，等待自動處理
    let b = fetch(20).await;
    a + b
}

// 落地法一：交給邊界——main 自己待在 Future 世界
#[tokio::main]
async fn main() {
    println!("{}", fetch_both().await);
}

// 落地法二（擇一）：在普通同步 main 裡，自己當場 block_on 結算
// fn main() {
//     let runtime = tokio::runtime::Runtime::new().unwrap();
//     let sum = runtime.block_on(fetch_both());
//     println!("{}", sum);
// }
# async fn fetch(n: i32) -> i32 { n }
```

`-> Result` 的 main 對上 `#[tokio::main]`、`match` 對上 `block_on`——同一個形狀，換了個世界而已。

## 重點整理

- `?` 和 `.await` 是**同一種東西**：黏在運算式後的小尾巴，把「包在特殊世界裡的值」拉出來，世界的規矩自動幫你串好
- 兩個世界形狀相同：`Result`＝「一個值**或**一個錯誤」、`Future`＝「一個值**只是還沒到**」
- 你寫的像直線程式，背後其實在把這些「包起來的值」串成一條計算；`?` / `.await` 是套用規矩的接縫
- `.await` 的規矩較複雜，所以 `async` 會把函數體改寫成狀態機（第 15 集）；兩者都會「傳染」（只能在世界裡面拉值）
- 作用最後一定要**結算**回普通世界，剛好兩條路：**頂到邊界交給框架**，或**自己當場結算**
- 對應關係：**`#[tokio::main]` ↔ 回傳 `Result` 的 `main`**（交給邊界）、**`block_on` ↔ `match`**（自己結算）；`block_on` 就是 sync→async 的橋（第 21 集）
