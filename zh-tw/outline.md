# Rust 教學影片大綱

## 第十一章：async / await 與非同步 runtime

1. **第一個 `async` 程式** — 直接用 Tokio 寫一個會回應 HTTP 請求的簡單 web server，讀者現在應該能依靠經驗理解 `async` 程式碼大概在做什麼了；伺服器用一個所有 request 共用的計數器，讓連進來的人看到「這是第 N 個 request」；介紹 Rust 的 `async` 可以使用在不同 runtime，而最常被使用的叫 Tokio。我們的程式頂層用的就是 Tokio 的 `#[tokio::main]`，重點解釋 `.await` 是「允許等待時做別的事」的意思，並讓學習者看到 `async` 真的能並行處理多個連線
2. **為什麼需要 `async`** — 說明 `async` 適合大量等待 I/O 的程式，例如網路伺服器、爬蟲、資料庫查詢、聊天室與背景工作；對比 OS `Thread`，強調 OS `Thread` 很吃記憶體。**並講清楚並行（concurrency）與平行（parallelism）的差別**：並行是「交錯切換、同時*處理*很多件事」，一個核心、一條執行緒也能做到（咖啡師一人輪流服務多桌的比喻）；平行是「多個 CPU 核心同一瞬間真的一起*執行*」。兩者是獨立維度——單執行緒 async＝有並行沒平行、純運算丟多核心＝有平行、多執行緒 runtime（Tokio 預設）＝兩者都有。`async` 提供的是**並行**（用少少幾條執行緒交錯推進大量等待中的工作），要不要平行由 runtime 用幾條執行緒決定。`async` 不是讓 CPU 計算變快，而是讓等待期間可以切去做別的事
3. **`async fn` 回傳的是 `Future`** — 用兩個方式顯示 `async fn` 的行為：(1) 呼叫 `async fn` 但不 `.await`，函數體裡的 `println!` 完全不執行，證明程式只是拿到一個還沒跑的 `Future`；(2) 故意把回傳值標成錯誤型別（`let x: () = f();`），讓編譯器直接回報 `expected (), found future`，當作「它真的是 `Future`」的確認。建立「呼叫 `async fn` 只是得到一個 `Future`」的心智模型，並點出新手最常見的坑：`Future` 是惰性的，不 `.await` 什麼都不會發生（只會拿到 `#[must_use]` 警告）；講惰性時順便連結第 6 章的**迭代器**——`Iterator` 也是惰性的（`.map`／`.filter` 要 `.collect`／`for` 才動），點出 `Future` 和 `Iterator` 是同一個「先描述、晚執行」的設計哲學
4. **`async` 是一種 effect** — 用 effect 的角度寫，**但全程不出現 monad 這個詞**：把 `?` 和 `.await` 講成同一種東西——黏在運算式後的小尾巴，把「包在特殊世界裡的值」拉出來，世界自動以某種規矩（Result 世界＝可能失敗、Future 世界＝還沒好）幫你串接；你寫的像一般程式，但背後編譯器其實在把這些包裝過的值串起來計算，每個 `?`／`.await` 是套規矩的接縫。`.await` 規矩比 `?` 複雜，會被大幅改寫成叫狀態機的東西，因此需要特別的語法；async 會「傳染」（只能在世界裡面拉值，一般函數裡不能直接 `.await`）。當然，作用最後一定要「落地」回普通世界，先用錯誤處理作為類比：要不就用「回傳 `Result` 的 `main`」，要不就被 `match` 掉。對應到 async——**`#[tokio::main]` ↔ 回傳 `Result` 的 main（交給框架在邊界結算），`block_on` ↔ `match`（自己在普通同步程式裡當場把 future 結算成普通值）**；並排兩個世界的兩條落地路線當範例。`block_on` 就是 sync → async 的橋
5. **async block** — `async { ... }` 可以在函數內直接建立一個 `Future`；對比 `async fn`：`async fn` 是定義可重複呼叫的 `Future` factory，`async` block 是當場建立一個匿名 `Future`；類比「具名函數 ↔ 閉包」。在 `Result` 世界，「當場一段能用 `?` 的區塊」用一個立刻呼叫的閉包 `(|| { ...? ; Ok(..) })()` 就能表達，不需新語法；但 Future 世界不能照搬寫成 `(|| { ... .await })()`，因為 `.await` 要把整段程式改寫成狀態機，而普通閉包只會編成普通函數、沒有「暫停」這回事，表達不了那種改寫——所以 async 才需要專屬的語法（等於叫編譯器「把這塊改寫成一個 `Future`」）。
6. **`Future` `trait` 與最簡單的 executor** — 介紹 `Future` `trait` 的正式定義。解釋為什麼 `poll` 的 `self` 能寫成 `self: Pin<&mut Self>`：因為 `Pin` 是很特別的型別——Rust 只允許一小撮「智慧指標類」型別放在 `self` 位置：`self`／`&self`／`&mut self`、`Box<Self>`、`Rc<Self>`、`Arc<Self>`、以及 `Pin<...>`；一般自訂型別不能這樣用，`poll` 能這樣寫正是沾了 `Pin` 被列入這份名單的光。寫一個 `loop` 空轉、不斷 `poll` `Future` 的 executor，然後秀出這個 executor 可以 drive Tokio 函數回傳的 `Future`。executor 用 `Box::pin` + `as_mut` 把 `Future` 接到 `poll`：`Box::pin` 是 `fn pin(x: T) -> Pin<Box<T>>`（把值放 heap 並釘住；目前先把 `Pin` 解釋成受限的指標就好）；`as_mut` 是 `fn as_mut(&mut self) -> Pin<&mut Ptr::Target>`，對 `Pin<Box<F>>` 即 `-> Pin<&mut F>`，正是 `poll` 的 `self: Pin<&mut Self>` 要的型別；`as_mut` 是借用，所以 `loop` 能反覆 `poll` 同一個 `Future`。在本集老實註明：到此為止（第 3–6 集）寫的 async（`async { 1 + 2 }`、`say_hello`、`make_coffee` 等）其實都「沒在等任何東西」——`poll` 會直接 `Ready`，純粹是拿來示範 `Future` ／ executor 機制用的，並非真正使用到 `async` 功能的程式；預告下一集的 `Delay` 才是第一個真的會 `Pending`、更像真正有 `async` 功能的 `Future`。最後，說明 executor 其實有很多種設計選擇：標準庫只定義 `Future` `trait`，怎麼實作 executor 留給 runtime 自由發揮，正因如此，Rust 才不內建 runtime，也才會有 Tokio、smol 等不同 runtime 的選擇：本章接下來會從最笨的版本一步步演進到接近真實 runtime 的實作
7. **手寫一個 `Delay` `Future`** — 手寫 `Delay` future，展示用前面的 executor 執行自己寫的 `Future`。明講動機：`Delay` 是用來「模擬一個需要花很久才完成的事件」的函數——真實的等待（網路、硬碟、資料庫）牽扯到 OS 之類的概念太複雜，不適合第一次就拿來認識 `Pending`，所以用最簡單的計時器當替身：到期前回 `Pending`＝事情還沒好、到期回 `Ready`＝完成。後面幾集都拿這個 `Delay` 當「需要時間才會好的事件」來研究 `.await`、並行與 `Waker`
8. **用 `.await` 等待 `Delay`** — 在 `async fn` 或 `async block` 中使用 `.await` 等待 `Delay`，並用最簡單的 `println!` 觀察執行進度：在每個 `.await` 前後都印訊息，讓讀者看到 `Future` 每次被 executor `poll` 時，會從上次暫停的地方繼續跑，直到遇到下一個還沒完成的 `.await` 才回 `Pending`；下一次被喚醒再 `poll`，才會越過那個 `.await` 繼續往下跑。範例放兩個以上的 `delay(...).await`，輸出會呈現「開始 → 等第一個 delay → 第一個完成後繼續 → 等第二個 delay → 第二個完成後繼續」的步進。最後再展示 `.await` 不會自動平行化程式：連續寫兩個 `.await` 會依序等待，下一集會解釋要並行要怎麼做
9. **手寫 join** — 自己寫一個 `Future`，把好幾個 `Future` 包起來，讓它們並行執行。**寫一個吃整個 `Vec`、用 `for` 迴圈 `poll` 到「所有」`Future` 都完成的 `JoinAll`**：`poll` 裡逐一推進每個未完成的 `Future`、做完換成 `None`，全部 `None` 才回 `Ready`。**join 時故意放一個「有多個 `.await`」的 `async` 區塊進去**，展示連「要 `poll` 很多次才會完成的 `Future`」也照樣被我們的 `JoinAll` 推進得動（`JoinAll` 完全不必特別處理，只管對每個 future 各 poll 一次）
10. **用 `Thread` 與 `Waker` 喚醒 executor** — 另外 `spawn` 一個 `Thread`，等事件完成後用 `Waker` `wake` executor；在這裡明確示範 `Waker` 是怎麼做出來的——實作標準庫的 `std::task::Wake` `trait`（`wake` 負責通知 executor「該再 `poll` 一次了」），再用 `Waker::from` 轉成 `Waker`。**並介紹 `unpark` 可以發生在 `park` 之前**：`poll` 回 `Pending` 到 `thread::park()` 之間有個空檔，若計時 `Thread` 剛好在這時 `wake()`→`unpark()`，executor 會不會「先被叫醒、才去睡」而睡死？不會——`unpark` 會留一張 permit，下次 `park()` 看到就立刻返回，所以 `wake` 不論落在 `park` 前或後都不會漏接（這也是直接用 `park`/`unpark`、不必自己刻旗標＋條件變數的原因）。後半明講：為每個等待的 `Future` 開一條 `Thread` 顯然不是好辦法——呼應第 2 集「`Thread` 很吃記憶體」，一萬個連線就一萬條 `Thread` 正是 `async` 想避免的問題；接下來幾集先把被 `wake` 的 `Future` 包成 `Task`，讓他能回到 executor 做出來的 ready queue，再用 reactor 處理「怎麼用少少幾條執行緒盯住很多 I/O」這個問題
11. **`spawn` 與 ready queue** — 開頭先破題導入 `Task` 這個新觀念：前幾集的 executor 手上永遠只有一個 `Future`、就在迴圈裡反覆 `poll` 它；但真實的 runtime 要同時養很多 `Future`，當某個 `Future` 的 `Waker` 喊「我好了」，如果只有一個裸 `Future`，executor 無從知道是哪一個、該排回哪裡——所以把每個 `Future` 包成一個 `Task`（＝「一個 `Future` ＋ 重新排程所需的隨身資料」：自己的 　`Future`、該排回哪條 ready queue、該叫醒哪條 executor `Thread`、一個避免重複入列的旗標），從此 executor 不再直接管 `Future` 而是管 `Task`，並預告 `spawn` 就是「把 `Future` 包成 `Task` 交給 executor」。接著才講機制：`Task` 的 `wake` 把 `Task` 放回 ready queue，再用 `Thread::unpark()` 叫醒睡著的 executor；executor 先清 ready queue，queue 空了但還有 `Task` 沒完成時，才用 `thread::park()` 睡覺。明講 `unpark` 只是一個「叫醒 executor」的門鈴，不指出是哪個 `Task` 好了；真正的 `Task` 是否該被 `poll` 了的資訊在 ready queue 裡。本集先只處理 `Future<Output = ()>`，讓讀者專心建立「`Task` `wake` → 進 ready queue → executor 醒來 → `poll` 該 `Task`」的資料流。範例：`spawn` 兩個不同延遲的 `Task`，展示各自到期時只把自己排回 queue。避免同一個 `Task` 重複入列的 `queued` 旗標用 `AtomicBool`（前面章節已教過 atomic），`schedule` 用 `queued.swap(true, ...)` 一步完成 check-and-set——只有讓 `false` → `true` 的那次 `wake` 會把 `Task` push 進 queue。executor 的入口叫 `block_on`：它把參數那個 `Future` 也 `spawn` 成一個 `Task`，然後跑這個連同先前 `spawn` 的所有 `Task`，直到全部完成。此外，還需要解釋 `Task` 中的 `Future` 欄位這次為何要 `+ Send`（前幾集沒有）：`Future` 被收進 `Task`，`Task` 又 `impl Wake` 兼任 `Waker`（註明雖然理論和實際上不一定要這麼做，但這邊能簡化實作），`Waker::from(Arc<Task>)` 又要求 `Task: Send + Sync + 'static`（舉例 `Waker` 需要能被 `Delay` `move` 到計時 `Thread` 上呼叫 `wake()`），連帶要求每個欄位（含 `Future`）都得是 `Send`——所以 `dyn Future` 要加 `+ Send`、外面包一層 `Mutex` 自動有 `Sync`；上一集是因為 `Waker` 構造比較簡單，因此我們當時不用思考這些 bound 的問題
12. **`spawn` 與 `JoinHandle`** — 開頭明講和第 11 集的差異：只在排程之上加三樣東西、其餘原封不動——(1) 新增共享的 `Shared<T>` 與 `JoinHandle<T>`(本身也是 `Future`)；(2) `spawn<T>` 從只收 `Future<Output = ()>` 變成收 `Future<Output = T>` 並回傳 `JoinHandle<T>`；(3) `block_on` 從回傳 `()` 升級成回傳傳進來那個 `Future` 的值 `T`。細節：executor 內部仍只存 `Future<Output = ()>`，`spawn<T>` 把 `Future<Output = T>` 包成標準 `Task`，完成時把結果放進 `Shared<T>`；`JoinHandle<T>` 本身不是 `Task`、不會進 ready queue，它只是被「等待者 `Task`」順帶 `poll` 的 `Future`；`.await` 時若結果還沒好，它用 `cx.waker()` 拿到**等待者 `Task` 自己的 `Waker`**（`JoinHandle` 沒有自己的 `Waker`）存進 `Shared<T>`、回 `Pending`，等**被 `spawn` 出去的那個 `Task`** 完成後，由它取出這個 `Waker`、`wake()`，於是把**等待者 `Task`** 排回 ready queue 並 `unpark()` executor。`block_on` 把傳進來的 `Future` 也 `spawn` 成 `Task`、保留其 `JoinHandle`，跑完後從它的 `Shared` 取出 `T` 回傳。強調這集不是 `Future` 直接通知 `Future`，而是完成的一方透過共享狀態與 `Waker` 喚醒等待的一方。範例： `spawn` 一個回傳 `i32` 的背景 `Task`，傳給 `block_on` 的 `Future` 在裡面 `.await` 這個背景 `Task` 的 `JoinHandle` 取得結果，自己再回傳一個值
13. **`mio`** — **專講 mio 主角，放在 reactor 前一集**。開頭先說明 reactor 在 runtime 裡的角色：executor 只管從 ready queue 拿 `Task` 出來 `poll`（跑 `Task`），不知道外部 I/O 發生了什麼；reactor 就是負責盯住所有 I/O 來源、哪個 ready 就 `wake` 對應 task 的那一塊（等事件），它不 poll `Future`、不是 `Task`，只注意外部事件來源——而 reactor 要「用一條 `Thread` 盯住很多 I/O 來源」靠的正是 mio，藉此帶出本集兩個主角：`mio::Poll`（睡著等 I/O 事件的地方，一條執行緒能同時盯很多 I/O 來源）、`Token`（事件來源的「名牌」）。**用一個範例讓讀者這集就看到 mio 跟 I/O 的關係**：把一個 `TcpListener` 用 `registry().register(&mut listener, token, Interest::READABLE)` 登記給 `Poll`，另一條 `Thread` 過 0.2 秒連進來，`poll.poll()` 被叫醒，`event.token()` 對上登記時給的名牌，接著 `accept()`（順帶帶出 `Interest::READABLE/WRITABLE`、`registry().register`、`accept`）。
14. **手寫 reactor** — 把第 11、12 集的喚醒模式接到真實 I/O。**核心訊息：executor 完全沿用第 12 集**（`Task` / `spawn<T>` / `JoinHandle<T>` / `Shared<T>` / `block_on<T>` 一行不改），這集只把「誰來 `wake`」從 `Delay` 計時 `Thread` 換成一條 reactor `Thread`。新增兩塊：(1) `Reactor`——跑在自己的 thread、睡在 `mio::Poll` 上等 I/O，醒來用 `Token` 找到對應 `Waker` 並 `wake()`；它不是 `Task`、不 `poll` `Future`。**跑在 executor `Thread` 上的 `Future` 與 reactor 用共享狀態溝通、不傳訊息**：三樣東西用 `Arc` 共用——`Registry`（`Future` 直接登記／取消 socket）、`AtomicUsize`（reactor 自分配 token）、`Mutex<HashMap<Token, Waker>>`（`Future` 在執行時寫入 `Waker`、reactor 取出來 `wake`）。(2) I/O future——`Accept` / `Read` 各自手寫 `impl Future`（不用 `poll_fn`）。注意：**先 `set_waker` 登記給 reactor，再試一次 `accept` / `read`**，成功回 `Ready`、`WouldBlock` 就回 `Pending`。順序刻意「先登記再試」避免 race。後面範例只 `.await` 少少幾個 request（`serve`：`accept` 一條連線 → 用 `for` 讀 3 個 request 印出來），把焦點留在 reactor。簡化：單一連線、`unwrap`、無計時器
15. **`async fn` 背後的狀態機** — 解釋 async fn 會被編譯器改寫成保存目前進度、區域變數與等待中 `Future` 的狀態機；`.await` 不是開新 `Thread`，而是把函數切成可暫停與恢復的幾段
16. **自我參照的 `Future`** — 從 `async` 狀態機中的 local variables 與跨 `.await` borrow 出發，說明 `Future` 可能變成自己欄位指向自己另一個欄位的結構，因此被 move 會有危險。**先用一段完全沒有 `async` 的普通程式（`let p2 = p1` 前後用 `{:p}` 印 `&p1`／`&p2`）讓讀者親眼看到「move 一個值位址就會變」**，點出這對一般值無所謂（Rust 會擋住誤用舊變數），再引導：當「值裡存了一個指向自己的位址」時，一搬家那個存起來的位址沒人更新、就會指向舊位置變成懸垂指標——而自我參照的 `Future` 狀態機正是這種值，藉此導入 move `Future` 是有風險的。**接著手寫一個最小 `Future`（自己寫 `poll`），直接示範「建立 → `poll` → move → 再 `poll`」**：用 `{:p}` 印 `self`，看到兩次 `poll` 的位址不同，證明這套流程真的做得出來、而且第二次 `poll` 時 `Future` 已在新位址；對沒有自我參照的 `Counter` 無害，但若 `self` 換成自我參照的狀態機，同一段流程就會踩到懸垂指標。**接著給一個真的編譯不過的例子做對比**：把同一套 `poll` ／ move ／ `poll` 套到一個「跨 `.await` 借用」的 `async fn`（`let r = &s; other().await; println!("{r}")`），會直接報 `the trait Unpin is not implemented`——因為 `Pin::new` 只收 `Unpin`，而自我參照 `Future` 不是 `Unpin`，Rust 在你還沒搬之前就把路擋住。對照 `Counter`（`Unpin`，隨便搬）凸顯防線：搬了不會壞的才放行、搬了會壞的連門都不給進（`Unpin` 留待第 18 集）
17. **為什麼 `poll` 需要 `Pin`** — 說明 `poll` 為什麼不是 `&mut Self`，而是 `Pin<&mut Self>`：`&mut self` 權力太大（能隨便 move），所以改用「閹割版的 `&mut`」——`Pin<&mut T>` 能改內容、但不准 move；強制方式是 `Pin` 不再洩漏普通 `&mut T`，只給你接受 `Pin<&mut Self>` 的方法。**接著介紹 `Pin` 各個能用的方法，幫讀者建立全貌**：唯讀靠 `Deref`（`Pin<P>` 永遠 deref 成 `&T`，附 `impl Deref` 簽名，呼應上一集 `&*self` 印位址）、`get_ref`、`as_ref`；重新借用成另一個 `Pin` 用 `as_mut`（呼應第 6 集 executor 反覆 `poll`）；建立一個 `Pin` 用 `Pin::new`（限 `Unpin`）/ `new_unchecked`（unsafe）/ `Box::pin`。**唯獨可變存取（`DerefMut` / `get_mut`，要 `Unpin` 才有）留到第 18 集**。最後點出 `Pin` 是型別層面的約定，`async fn` 一般使用者幾乎不會直接碰到，細節主要給寫底層 future／runtime 的人
18. **`Unpin` 與 `get_mut`** — 說明 `Unpin` 是 `auto trait`：普通手寫 `Future` 若欄位都可安全 move，通常會自動是 `Unpin`；但 `async fn` / `async` block 產生的 `Future` 不能直接假設是 `Unpin`；介紹 `T: Unpin` 時 `Pin<&mut T>` 可以用 `get_mut` 回到普通 `&mut T`，以及哪些函數或方法會要求 `Unpin`
19. **`pin!`** — 介紹 stack pinning；說明 `pin!` 會在目前 scope 建立 pinned local，回傳 `Pin<&mut T>`，適合不想 heap allocation，不需要把 pinned `Future` 傳出 scope 的情境。**並解釋為什麼 `pin!` 一定要是巨集、不能是函式**：stack pinning 的本質是「在呼叫端的 stack frame 放一個 local，再交出借用它的 `Pin<&mut T>`」；若做成函式 `fn pin<T>(value: T) -> Pin<&mut T>`，`value` 住在函式自己的 frame，函式一返回 frame 就回收、回傳的借用立刻懸垂（borrow checker 也禁止回傳指向自身 frame local 的參考）。巨集則就地展開到呼叫端，能在你的 scope 宣告 local（壽命跟周圍 scope 一樣長）、再用同名 shadow 蓋掉可移動的原值（封掉 move 後門）——附簡化展開（`let mut fut = fut; let fut = unsafe { Pin::new_unchecked(&mut fut) };`）。對照 `Box::pin` 把值放 heap、交出所有權（活得比這次呼叫久），所以能當普通函式；差別就在「stack 借用 vs heap 擁有」
20. **async 遞迴** — 一個 `async fn` 直接呼叫自己會編譯失敗，因為 `Future` 的大小會變成無限大；解法是把遞迴呼叫包成 `Box::pin(...)`，呼應第五章「遞迴型別需要 `Box`」；`Box::pin` 是「想把不同的 `Future` 塞進同一個 `Vec`」或「`match` 兩個分支回傳不同 future」時的解法，呼應第九章的 `Box<dyn Fn>`
21. **回到 Tokio** — 從前面手寫的 executor 回到 Tokio，說明 `Runtime::block_on` 只是把傳給它的 `Future` 在目前呼叫的 `Thread` 上跑到完成（這也是從 sync 跨進 async 的橋樑，呼應第 4 集），因此不需要 `Send` 或 `'static`；**並明確點出和第 11～14 集手寫 `block_on` 的語意差異**：手寫版會等 ready queue 裡「所有 `Task` 都完成」才回傳（教學版好觀察的簡化），Tokio 的 `block_on` 則是「傳給它的 `Future` 一完成就回傳」、不等其他 `spawn` 的背景 `Task`（未完成的留在 runtime 上直到 `Drop`）——「跑完整批」對「跑完我指定的這一個」；相對地，`tokio::spawn` 會把 `Task` 交給 runtime 排程，多執行緒 runtime 可能在不同 `Thread` 之間移動 `Task`，因此 `Future` 與 output 都需要 `Send + 'static`；在這裡帶出新手最常見的編譯錯誤——在 `.await` 期間持有非 `Send` 的值（`Rc`、`RefCell` 等）會讓整個 `Future` 變成 `!Send`，於是不能 `spawn`，並示範解法（在 `.await` 前先把值 `drop` 掉、或用 `{}` 縮小其 scope）；同時介紹 `#[tokio::main]` 預設使用多執行緒 runtime，但可以用 `flavor = "current_thread"` 改成單執行緒，或用 `flavor = "multi_thread", worker_threads = N` 指定 worker 執行緒數
22. **`join!`** — 介紹 `join!` 會在同一個 `Task` 中同時等待多個 `Future`，把它們 multiplex 在目前的 `async` context 裡，而不是像 `spawn` 一樣交給 runtime 變成獨立 `Task`；說明 `join!` 會等所有 branch 都完成，因此適合固定數量、生命週期綁在目前函數內的並行 I/O。**講完功能後，解釋為什麼 `join!` 要是巨集而不是函式**：它要吃「任意數量、各自不同型別」的 future，並回傳一個形狀對應的 tuple（`join!(a,b,c) -> (A::Output, B::Output, C::Output)`）；Rust 函式不能 variadic、更無法「參數個數任意＋每個型別不同＋回傳形狀隨之變化的 tuple」，只能用巨集在編譯期按實際傳入的 branch 生成程式碼（對照第 9 集 `JoinAll` 是「同型別、動態數量」的 `Vec<F>`，用一般 struct／函式就行；`try_join!` 同理也是巨集）。同時對比 `try_join!` 遇到第一個 `Err` 會提早回傳，以及 `join!` 的並行不等於 CPU 平行計算，若某個 branch 長時間不 yield 或做 blocking work，其他 branch 仍會被拖住；由此帶出「不要在 `async` `Task` 裡做 blocking work」的原則，以及解法 `tokio::task::spawn_blocking`
23. **`AsyncRead` 與 `AsyncWrite`** — 介紹 async 版 byte I/O，說明 `read` / `write` 只承諾嘗試推進一次並回傳本次處理的 byte 數，不保證讀滿或寫完；透過 `AsyncReadExt` / `AsyncWriteExt` 說明 `read_exact`、`write_all` 等 helper 的便利。**並藉這些 helper 第一次帶到「取消（cancellation）」**：future 是惰性的，把它 drop（不再 poll）就等於取消這個 async 工作——這是 async 特有、thread 做不到的乾淨喊停；而 `read_exact` 這類「跨多次推進、累積中間狀態」的操作被中途取消會遺失已讀資料，因此**不是 cancellation safe**，不該放進「會中途丟掉它」的地方（最典型就是下一集 `select!` 的 branch）。把 buffer ownership、partial read/write 與前面的 reactor/runtime 模型連起來
24. **`select!`** — 介紹 `select!` 會同時等待多個 branch，只要第一個完成就執行對應 handler，其他 branch 會被 drop——而「drop 一個 future＝取消」正是上一集（第 23 集）帶到的 cancellation，`select!` 是**最常製造取消的地方**（標題不再寫 cancellation，因為概念已在第 23 集introduce）；說明 `select!` 適合 timeout、接收多個 channel、等待 shutdown signal、race 兩個操作等情境；補充 branch precondition、`else`、預設公平性與 `biased;`，並提醒在 loop 中使用 `select!` 時要注意 cancellation safety（呼應第 23 集 `read_exact` 的例子），避免把不可安全取消的 future 放進會反覆被 drop 的 branch
25. **`Semaphore` 與 backpressure** — 介紹 semaphore 是用 permits 表示容量的同步原語，適合限制同時下載數、同時開檔數或同時進入某段流程的 task 數；說明 acquire permit / drop permit 的 RAII 模型，以及 bounded channel、connection pool、worker pool 等 backpressure 都可以用「容量有限，滿了就等待 permit」的角度理解
26. **`mpsc` channel：async 的工作佇列** — 介紹 multi-producer single-consumer channel 作為 async task 間最常見的工作佇列；從 bounded `mpsc` 的容量限制接續上一集的 backpressure，說明 `send().await` 為什麼會等待、receiver 如何形成 worker loop，以及如何搭配 `select!` 同時等待工作與 shutdown signal
27. **`oneshot`、`watch` 與 `broadcast`** — 用訊息拓樸區分不同 channel：`oneshot` 適合一次性回傳結果，`watch` 適合只關心最新狀態或 shutdown flag，`broadcast` 適合多個 receiver 都要看到每則訊息；強調選 channel 時先問「訊息有幾個 sender、幾個 receiver、要保留所有訊息還是只要最新狀態」
28. **`async` 的 `Mutex`、`RwLock` 與 `Notify`** — 介紹當 message passing 不適合時如何處理共享狀態：`Mutex` / `RwLock` 保護 shared mutable state，lock scope 要短，避免拿著 lock 等 I/O；`Notify` 則是不帶 payload 的喚醒 primitive，適合搭配共享狀態使用，提醒它不是 queue，多次通知可能合併
29. **`Stream`：async 版的 iterator** — 介紹 `Stream` 是 async 版 iterator，`next().await` 代表等待下一個 item；從 channel receiver、line-based input、週期性事件等例子建立「連續產生 async values」的心智模型，並使用 `tokio_stream` / `futures::StreamExt` 展示基本 combinators
30. **`FuturesUnordered`：大量動態的 Future** — 對比 `join!` 適合固定少量、異質 futures，而 `FuturesUnordered` / `for_each_concurrent` 適合大量、動態產生的 futures；說明它像 completion stream，哪個 future 先完成就先產生結果，適合爬蟲、批次請求與限制 concurrency 的工作流
31. **測試 `async` 程式** — 介紹 `#[tokio::test]`：自動幫測試函數套上 runtime，不用自己 `block_on`；並說明 `tokio::time::pause()` / `advance()` 可以手動推進時間，讓牽涉 timeout、延遲的測試變得 deterministic、不必真的空等；呼應前面 `cargo test` 的章節
32. **Tokio 以外的 runtime** — 說明 Rust 標準庫只定義 `Future` 等語言層抽象，並沒有內建 async runtime；runtime 通常包含 executor、reactor、timer、I/O 與 task API。以 Tokio 為主流對照 smol、已停止維護的 async-std、以及 monoio / glommio / Embassy 等特化 runtime，說明不同 runtime 的 API、I/O driver 與 task model 可能不同，寫 library 時應盡量區分 runtime-agnostic 的 Future 組合邏輯與 runtime-specific 的 I/O、timer、spawn
33. **`async` 閉包** — 介紹 Rust 的 async closure `async || { ... }`，對比 `|| async { ... }` 的 capture 與 lifetime 差異；說明 `AsyncFn`、`AsyncFnMut`、`AsyncFnOnce` 讓高階函數可以直接表達「接受一個可被 await 的 closure」，並回顧過去常見的 `Fn(...) -> Fut where Fut: Future` 寫法

## Optional 進階補充

以下主題超出基礎教學範圍，供進階學習者參考：

- **Lifetime Bound** — `'a: 'b` 表示 `'a` outlives `'b`；`T: 'a` 表示 T 內的所有參考都活得過 `'a`；當函數或 struct 有多個 lifetime 參數時，指定它們之間的關係
- **HRTB（Higher-Ranked Trait Bounds）** — `for<'a> Fn(&'a str) -> &'a str`；當 lifetime 不能在函數簽名上寫死時使用；thread::scope 的簽名完整解讀
- **Variance** — covariant / contravariant / invariant；`&'a T` 對 `'a` 是 covariant，`&'a mut T` 對 `T` 是 invariant；用 `PhantomData` 控制 variance
- **GAT（Generic Associated Types）** — associated type 帶泛型參數：`type Item<'a> where Self: 'a;`；Rust 最接近 HKT 的功能；主要用於 lending iterator 等 library 設計
- **RPIT 進階（`use<...>`）** — Return Position Impl Trait 的 lifetime 捕捉行為；`impl Trait + use<T>` 精確控制捕捉哪些泛型參數和 lifetime；Rust 2024 edition 的行為變更
